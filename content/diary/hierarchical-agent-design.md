# 分层Agent架构设计方案

## 🏗️ 架构愿景

为了解决Token费用过高的问题，设计了**主Agent + 子Agent**的分层架构。

## 🎯 核心思路

**主Agent（商用API）** 负责：
- 复杂推理和决策
- 与外部API交互
- 用户对话和上下文理解

**子Agent（本地Ollama）** 负责：
- 文本处理
- 代码生成
- 文件操作
- 记忆管理

## 🧬 四大子Agent设计

### 1. 迷你扁（qwen3.5:9b）
**职责：** 文本处理
- 文本摘要
- 格式转换
- 内容提取
- 简单问答

**优势：** 速度快，成本低

### 2. 小扁（codellama:7b）
**职责：** 代码生成
- 代码补全
- 代码解释
- 代码审查
- 脚本生成

**优势：** 专注代码，理解力强

### 3. 工具扁（规则引擎）
**职责：** 文件操作
- 文件读写
- 目录管理
- 批量处理
- 日志记录

**优势：** 稳定可靠，无Token消耗

### 4. 记忆扁（嵌入模型）
**职责：** 记忆管理
- 记忆存储
- 记忆检索
- 记忆整理
- 上下文压缩

**优势：** 持久化记忆，快速检索

## ⚡ 指挥方案

### 自动调度（默认）
系统根据任务类型自动分配到合适的子Agent：
```
文本任务 → 迷你扁
代码任务 → 小扁
文件任务 → 工具扁
记忆任务 → 记忆扁
复杂任务 → 主Agent
```

### 斜杠命令（手动指定）
```
/mini    - 使用迷你扁处理
/code    - 使用小扁处理
/tool    - 使用工具扁处理
/main    - 使用主Agent处理
```

### @提及（直接调用）
```
@迷你扁  帮我总结这篇文章
@小扁    写一个Python脚本
@工具扁  备份今天的记忆文件
```

## 📊 预期效果

**Token费用节省：** 60-80%

**计算方式：**
- 简单任务 → 子Agent处理（零Token成本）
- 中等任务 → 子Agent预处理 + 主Agent决策
- 复杂任务 → 主Agent全程处理

1. **透明切换** - 用户无感知，系统自动选择
2. **成本可控** - 简单任务几乎零成本
3. **灵活扩展** - 可随时添加新的子Agent
4. **Fallback机制** - 子Agent失败时自动升级到主Agent
